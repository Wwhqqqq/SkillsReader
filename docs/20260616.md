## 一、产品说明（自然语言）

### 1.1 SkillRadar 是什么

SkillRadar 是一个面向 Agent Skills 生态的**自动化情报系统**。它每天从 skills.sh、GitHub 等公开渠道持续抓取新出现的 Skill，把它们清洗、分类、打分后写入数据库，再按一套可解释的精选规则，把最值得关注的条目整理成**双榜日报**，通过如流机器人推送给指定同事。

你可以把它理解为一个「Skill 领域的编辑室」：后台记者（采集器）全天在外面跑新闻，编辑部（算法）从两堆稿子里各挑 8 条最有价值的，排版成一张易读的消息卡片，定时发到你的如流单聊或群聊。

它的目标不是把所有 Skill 都推给你——那样信息噪音太大——而是帮你回答两个日常问题：**今天有什么新东西值得看？** 以及 **现在全库里最火、最值得参考的是哪些？**

---

### 1.2 一天怎么运转：从早推到晚归档

SkillRadar 按**上海时区**运转，默认每天有两个自动节点：

**早上 10:00**，系统执行「早推」。它会先拉取过去 24 小时内各渠道的新数据，更新数据库里的安装量、星数等指标，把真正首次发现的高质量 Skill 入库；然后按双榜规则各选出最多 8 条，拼成一条 Markdown 双表格消息，推送到如流。**早推不会归档**，也就是说，当天日榜池里的 Skill 仍然保留在「今日新增」候选里，方便你白天继续手动测试、补推或预览。

**下午 18:00**，系统执行「晚推」。流程与早推相同：采集、入库、双榜精选、推送。但晚推结束后会多一步——**把当日 cycle 的 Skill 归档入库**。归档不是删除，而是给这些 Skill 打上一个「今日 cycle 已结束」的标记。归档之后，它们会从「今日新增」候选池移出，进入「当前总榜」候选池；Skill 本身仍是活跃状态，后续安装量变化会继续更新。

这样设计的好处是：**每一天都是一个新的循环**。早上 10 点打开如流，你看到的是「今天新发布/新发现」和「当前总榜」两张表；到了晚上 6 点，当日这批「今日新增」会被正式收档，第二天早推时，日榜又是全新的一批，不会和昨天重复纠缠。

**手动操作与定时互不干扰。** 你在白天随时可以用调试台预览、手动跑一次完整流程、或补推某天的日报——这些操作**都不会触发归档**。只有晚档定时任务（或你显式执行归档命令）才会结束当日 cycle。因此，测试环境和生产节奏可以并存：白天随便试，晚上系统自动收口。

---

### 1.3 双榜日报：两条线、各选八条

推送到如流的消息不是单一列表，而是**两个板块**：

**今日新增**，回答「今天有什么新东西」。候选来自「源站发布日期为今天、且尚未被当日归档」的活跃 Skill。编辑标准偏向「新闻价值」：质量是否过关、描述是否充实、分类是否清晰、是否官方首发、地域与场景是否多样。最终最多展示 8 条，带 🆕 标记。

**当前总榜**，回答「现在什么最火」。候选来自「发布日期早于今天、或已被归档、或尚无发布日期」且安装量达标的 Skill，在全库中取前几百条做池子，再按总榜专用打分规则精选最多 8 条。这里更看重安装量与持续性热度，但也设有质量下限，避免纯数字灌水。

两个榜单**独立选题、顺序执行**：先定总榜 8 条，再定日榜 8 条；日榜会自动排除「历史上已在总榜出现过的」以及「本批总榜刚选中的」Skill，避免同一条 Skill 在两个表里重复抢版面。若极少数情况下仍出现交叉，消息里会用 🔁 标注，方便你辨认。

消息以 Markdown 双表格呈现，包含序号、Skill 名称、来源仓库、安装量；官方 Skill 带 ⭐，新增带 🆕。你可以配置是否展示置信度等调试信息，默认面向阅读体验做了收敛。

---

### 1.4 数据从哪来、怎么进门

SkillRadar 目前接入了多条采集线路，按优先级并行工作：

**skills.sh** 是主数据源，覆盖 hot、trending、top 等榜单板块，能拿到安装量、描述、链接等一手信息。**GitHub Code Search** 按 SKILL.md 等关键词扫描公开仓库，补足 skills.sh 未收录的长尾。**GitHub Watch** 持续监控一批官方与社区核心仓库（如 anthropics/skills、vercel-labs/agent-skills 等），第一时间发现结构变更或新 Skill 目录。

**Gitee、Awesome 列表、MCP Market** 等线路已在架构上预留，部分仍为占位状态，默认不影响主流程。

每条抓到的原始记录会经过统一格式化、指纹去重、规范校验、规则分类（地域 + 场景关键词）、质量打分。分数达到入库门槛（默认 40 分）的记为活跃 Skill 写入 MySQL；不达标的过滤存档，不进入推送候选，但保留追溯能力。首次入库时，系统还会尝试从 GitHub 补全**源站发布日期**，这是区分「今日新增」与「总榜老将」的关键依据——没有发布日期的老条目，在补全之前会归入总榜侧处理。

---

### 1.5 如流怎么收消息

SkillRadar 已对接**如流开放 API**（非简单 Webhook 占位），支持：

- **单聊推送**：默认推给指定 UUAP 用户（如个人测试账号），适合联调与内测；
- **群聊推送**：需显式开启群权限开关，并配置群 toid，避免未经审批向群发送；
- **告警通道**：采集失败、推送失败等异常走单独的告警路径，默认只打日志或发单聊，不打扰群。

日报走 `send_digest`，告警走 `send_alert`，两者目标解耦。群聊消息支持 TEXT 或 MD 类型；单聊为文本。你可以通过环境变量切换推送目标（仅单聊 / 仅群聊 / 两者），官方新发 Skill 的即时告警默认关闭，避免测试阶段消息泛滥。

### 1.6 历史、预览与调试

系统会把每次成功推送的日报**存档到数据库**（`daily_digests` 表），保留近 15 日的精选结果、推送状态与选型元数据。你可以：

- 列出近 15 日已推日报索引；
- 查看某一天的存档正文，或按当前数据库**重新生成**再对比；
- 对历史某一天**补推**到如流，或推送一份「近 N 日索引表」方便回顾。

本地提供 **Streamlit 调试台**（默认端口 8501），包含：数据概览、双榜诊断（池子规模、排除规则、跨榜重叠、分项得分）、历史日报浏览、如流消息预览（不真正发送）、Skill 浏览与筛选、一键采集/试跑、环境配置指南。推荐在对接如流前，先在调试台把正文和 payload 核对满意，再开真实推送。

---

### 1.7 健康检查与运维节奏

除早晚推送外，系统默认**每小时**做一次健康检查：数据库连通、通知器配置、最近一次采集是否成功、GitHub Token 是否可用等。结果可写入日志，失败时退出非零状态码，便于 Cron 或监控系统感知。

本地 Mac/Linux 可用一条命令安装用户级 crontab（早 10、晚 18、每小时 health）；Docker 部署侧亦有对应 crontab 模板。手动试跑、补推发布日期、单独归档等能力通过命令行暴露，不依赖图形界面。

---

### 1.8 当前边界与已知限制

以下能力**尚未**作为产品承诺交付，或仅部分就绪：

- LLM 分类兜底（规则分类为主，大量条目可能暂为「未分类」）；
- Gitee / Awesome 真实采集（架构已有，逻辑待补）；
- MCP Market 采集（默认关闭）；
- 官方新发即时告警（默认关闭）；
- 远程 SKILL.md 深度校验；
- 人工分类反馈闭环；
- 多群订阅、个性化订阅、周报月报等产品化能力。

另：历史冷启动阶段，若大量 Skill 的发布日期尚未从 GitHub 补全，总榜池可能偏少、日榜池按发布日划分的结果会与「首次发现日」观感不同——可通过批量补全发布日期逐步收敛。

## 三、关键配置语义（无代码速查）

| 配置项 | 默认值 | 含义 |
|--------|--------|------|
| `use_publish_date_for_pools` | true | 日榜按源站发布日、总榜按发布早于今日或已归档 |
| `evening_archive_after_push` | true | 晚推后自动归档当日 cycle |
| `morning_push_cron` | 0 10 * * * | 早推时刻 |
| `evening_push_cron` | 0 18 * * * | 晚推时刻 |
| `leaderboard_history_days` | 15 | 日榜排除「历史总榜」的回溯天数 |
| `enable_official_alerts` | false | 官方新发是否即时告警 |
| `RULIU_ALLOW_GROUP` | false | 是否允许向群聊发送 |
| `RULIU_NOTIFY_TARGET` | dm | 推送目标：单聊 / 群聊 / 两者 |

## 五、一句话总结

**SkillRadar 每天 10:00 和 18:00 自动采集 Agent Skills 生态数据，各推送一张「今日新增 + 当前总榜」双榜日报到如流；18:00 推送完成后将当日 Skill 归档收档，次日日榜重新开局；白天可随时手动预览和试推，不影响当日测试节奏。**
